## 文档骨架

所有项目沿用同一套外层骨架，技术章节按项目实际内容增减：

1. 产品形态与项目背景：先用产品图建立具体认知，再讲现场问题、整机组成、实际工作方式、职责和交付阶段。
2. 软硬件详细设计资料：主手稿在开篇提供 PDF 预览和 Word 下载，不展开整份说明书。
3. 学习准备：把理解本项目必须掌握的知识直接讲清，外部资料只做事实核查。
4. 30 秒与 90 秒讲法：先给短版本，再给完整主线。
5. 从需求到交付：先给一段 60～90 秒面试回答，再按项目实际阶段说明输入、动作、输出、评审和回退条件。
6. 架构、执行域、数据流和职责边界：先分清谁采、谁算、谁传、谁恢复。
7. 技术问答：按真实数据流和项目模块组织，不按简历条目机械拆章，也不使用学习优先级标签做可见标题。
8. 实战调试案例：案例边界来自项目资料，缺失的机制、参数和证据链按项目底稿补全，并形成完整因果链。
9. 设计取舍与反向追问：解释为什么这样做，以及哪些边界不能夸大。

技术章节通常按“设备与执行域 → 采集与驱动 → RTOS 与数据流 → 算法与状态判断 → 对外通信 → 维护升级 → 故障诊断”展开。项目边界必须有资料支撑；边界内缺失的机制和数值按项目底稿完成工程补全。例如项目没有 BLE 或 Bootloader，就不要为了结构完整硬加章节。

如果处理器、执行域或主辅链路仅靠文字不容易分清，在 30/90 秒介绍之后、技术问答之前放一张整机图。复杂项目还要从软硬件设计内容中提取任务、协议、状态、资源和调试机制，按 [engineering-figures-imagegen.md](engineering-figures-imagegen.md) 规划额外配图并插入相应章节；简单项目不强制凑图。

“从需求到交付”不能写成通用教材目录。每个阶段至少写清本项目的输入、执行动作、输出物、评审准入和失败后的回退位置，并说明候选人在职责范围内做了什么。完整手稿必须有一道能够直接回答面试官的流程题；阶段超过五个时，优先增加一张纯 ImageGen 工程流程图，正文保留精确阶段名称和产物说明。

## 开篇首屏

设备、可穿戴、机器人、仪器和终端项目的首屏固定为：

```text
项目标题
一级标题：产品形态与项目背景
ImageGen 产品图
图注
项目背景与实际工作方式
一级标题：软硬件详细设计资料
```

- 产品图放在第一个一级标题正下方。不要先放学习准备、抽象架构、目录式说明或问答。
- 有真实素材时优先用真实素材；没有实物素材但需要让学员看懂具体形态时，用 `product-mockup` 生成写实产品展示图。
- 图中呈现项目实际需要的设备本体、佩戴/安装方式、配套手机/上位机/网关和环境关系，不把普通单品照当作整套系统展示。
- 生成图不承担实物取证功能。可见文档无需标注“概念图”，但正文不得称它为样机照片、路测照片或测试证据。
- 涉及驾驶、医疗操作、高压设备和生产现场时，默认画安全、静止或受控状态；不让人物边驾驶边操作电脑，也不制造其他不安全动作。

软硬件资料紧跟项目背景，不放在问答之后。嵌入式软件岗位默认顺序为软件说明书在前、硬件说明书在后；硬件岗位反过来。每份资料依次放名称、范围导读、PDF 预览、同版本 Word 下载。PDF 和 Word 的标题、版本、日期必须一致；写回后实际打开预览确认可访问，不能只检查附件 token 存在。

## 问答单元

每个问题保持以下结构：

```xml
<h2>Q1：问题</h2>
<p><span text-color="rgb(140,140,140)">真实面经：代表公司等 N 份复盘出现。</span></p>
<p><span text-color="rgb(100,37,208)">完整、正式、可直接讲给面试官听的回答。</span></p>
<p><b>补充理解：</b>只解释真正容易混淆、且完整回答不适合展开的内容。</p>
```

- 普通技术问答没有真实面经来源时省略灰色来源行；进入“真实面试问题与完整回答”章节的问题和嵌套追问必须保留该行。
- 整个回答正文使用紫色文本，不另加“可直接口述回答”“标准答案”等标签。长回答默认不整段加粗，避免像培训提示框。
- 一个问题只给一份答案，不生成简版、进阶版或按时长拆分的多个版本。回答先说结论，再讲关键机制、边界或证据。普通问答能一段说清就不要拆段；调试案例最多用三段分别承载“现象与范围、证据与根因、修改与验证”。
- 黑色“补充理解”按需添加，不是每题必配。通常一到两小段即可。
- 技术路线、30 秒介绍、90 秒介绍、所有问答和三个调试案例的完整回答都执行相同紫色规则；一个回答跨多个连续段落时，每段都必须为紫色。
- 术语第一次出现且可能卡住学员时，用一句白话解释；不要重复教材定义。
- 数据手册、协议规范和工具链接放在补充理解，不塞进完整回答。
- 调试案例按“现象 → 排查范围 → 关键证据 → 根因 → 修改 → 验证”自然讲述，但不要把这六项写成机械标签。标题下先用一句话交代根因和处理结论，让学员在十秒内找到现象、根因、修改和验证。
- 七项案例结构只用于内部审查；“注入设定、拟执行、判断条件、证据成立时、确认后、验收计划”等制作状态不得进入学员成稿。
- 案例主回答通常只保留二至四个决定性数字，详细波特率、字段、任务名和压力测试矩阵放进黑色补充理解。不要用一整屏同权重紫色文字迫使读者自己提炼主线。

## 语言要求

- 写具体对象、动作、时序和结果，例如“HT 和 TC 连续到来，二值信号量只保留一次唤醒”，不要写空泛的“提升系统能力”。
- 句子短，主语明确。读出声时应像工程师在复盘项目，不像背报告。
- 芯片、接口、参数和指标先服从项目口径底稿。量化结果只有得到日志、测试记录、可复现测量或用户明确确认时才称为真实场景结果；其余写作设计参数或未核实数据。缺失的技术参数由执行者按官方约束和成熟工程实践确定最终值，完成计算后写入底稿并核对全篇。命名客户、认证、日期、团队规模和个人职责不能由工程补全推出。
- 技术回答要能承受追问。通信校验、业务正确性、缓冲所有权、掉电恢复等概念不要混为一谈。
- 不堆“首先、其次、最后”，不使用“核心能力、关键抓手、形成闭环、进一步赋能”等抽象套话。
- 禁用“面试官意图”“核心亮点”“标准答案”“自测题”等栏目。
- 不使用表情符号、装饰性提示框、花哨分栏和无信息价值的强调。

## 真实面试问答与原题档案入口

有可靠真实面经时，主稿末尾增加“真实面试问题与完整回答”。它是学员理解项目后的考前突击章节，按项目技术主线组织，不按公司逐家罗列，也不预设题数。

每道主问题使用当前章节的主问题标题层级；只有真实出现、确有价值并且需要另一套回答逻辑的追问，才降一级嵌套。不要显示“连续追问”等解释性标题，也不要放一串没有答案的问题。主问题和嵌套追问都按问答单元排版：标题、灰色来源行、一份完整紫色回答，以及按需出现的黑色补充理解。

同一回答逻辑的不同问法合并成一道问题。公司或行业只在真实约束会改变答案时形成独立追问；否则只在灰色来源行中列出代表公司和可复核的出现次数。来源次数按不同面经记录计算，详细链接保留在独立原题档案。

独立原题档案按公司、岗位、轮次或记录 ID 保存全部在范围内的原题、原顺序和来源链接，不筛选、不润色、不去重，也不补生成答案。主稿另设一个紧凑的“真实面经原题档案”入口，只放一小段用途说明和一个可访问的文档卡片或书签，不再次复制原题列表。具体范围、归并和计数按 [real-interview-evidence.md](real-interview-evidence.md) 执行。

## 图像规则

所有自定义解释图必须按 [engineering-figures-imagegen.md](engineering-figures-imagegen.md) 使用纯 ImageGen 生成、筛选和无损插入。禁止纯文本路线图、ASCII 图、代码块、Mermaid、SVG、画板和本地重绘。用户提供或权威来源的实物照片、原理图、波形、日志、界面截图可以作为来源图；没有实物素材时，机制图使用抽象工程关系，产品总览与 HUD 章节允许生成克制的产品展示图。不要在可见文档中标注“概念图”“非实物”“真实样机照片”，也不要在口述中把生成图说成现场照片。

优先为以下内容配图：多处理器/跨端边界、跨任务数据所有权、DMA/队列路线、协议分帧、状态与恢复、内存/功耗约束和故障证据路径。单一结论能用两句话说清时不额外画图；复杂项目不能只放一张总架构图。

每张图都要有文档外部的编号和标题。图中的芯片名、接口名、方向、单位、状态名必须和正文一致。错误候选直接拒绝并重新生成，不能本地修图或靠正文解释错误。

## 飞书与 DOCX 的交付关系

- 迭代阶段以飞书文档为唯一活稿。问答、图片、补充理解和顺序都在飞书中更新。
- 用户提供飞书链接或明确要求飞书时，只把飞书主手稿作为活稿；只有用户明确要求导出、快照或主手稿 DOCX 时才从同一 revision 生成。软硬件设计说明书是独立附件，可同时保留同版本 DOCX 和 PDF。
- 设计说明书在飞书中采用“短导读 + PDF 预览 + Word 下载”的方式。不要把几十页说明书全文复制到主手稿，也不要只给一个本地文件名而没有可访问附件。
- 设计说明书位于产品形态与项目背景之后、学习准备之前。嵌入式软件岗位默认先软件后硬件，其他岗位按相关性排序。
- 飞书可用时，DOCX 交付后如需修改，先回到飞书更新，再重新生成一个新版本的 DOCX。不要分别修改两份文件。
- 飞书可用时，旧 DOCX 只保留归档，不再继续编辑。离线回退时只把最新版本 DOCX 作为临时活稿，每次改版归档上一个版本。文件名使用 `项目名_项目面试手稿_vX.Y_YYYYMMDD.docx`。
- 飞书暂时不可用时，以 DOCX 作为临时主稿并记录版本；飞书恢复后将最新 DOCX 建成飞书活稿，再按上述单活稿规则迭代。用户明确只要一种格式时，按用户要求交付并说明没有生成另一份。

## 写入前检查

1. 飞书可用时，读取最新 revision、目录和待改章节，任何写操作都以最新 block id 为准；离线回退时，记录最新 DOCX 的文件名、版本和修改时间，并只在该版本上继续。
2. 对照简历、设计文档、代码或用户补充材料，确认项目边界、处理器数量、总线用途和实际成熟度；资料未说明成熟度时，只按小批量可交付目标完成工程设计，不声称已经量产。
3. 检查已有图片中的接口、参数、方向和状态是否已经过时。
4. 按 `clarification-gate.md` 处理不确定项：可查证事实自行查证，技术机制、路线和参数由执行者裁决；只有仍会改变个人职责、产品物理形态或项目边界的冲突才逐个询问。
5. 确定本次只新增、替换或移动哪些块，避免顺手改动无关章节。

## 写入后检查

1. 重新读取目录和所有改动章节，检查顺序、重复块、标题层级和段落位置。
2. 对具有物理形态的项目，确认打开文档第一屏就能看到产品图；项目背景之后立刻可见软件/硬件 PDF 预览和 Word 下载，学习准备与问答在其后。
3. 确认技术路线、30/90 秒介绍、全部问答和三个案例的完整回答均为紫色，且没有“可直接口述回答”标签；每个问题只有一份答案，补充理解为普通黑字，没有混入回答段。
4. 顺着整机数据流通读一遍，检查执行域、缓冲所有权、协议边界、升级状态和调试证据能否互相对上。
5. 逐张核对纯 ImageGen 图文，特别检查接口名、单位、箭头、协议时序、DMA 事件和存储分区等容易误导的内容；确认图片位于对应章节、旧图和文本路线图已删除。
6. 把每段紫色回答实际读一遍；不顺口、句子过长或听起来像报告时，继续压缩。
7. 对每个调试案例执行十秒扫读：不细读全文也能指出现象、根因、修改和验证；做不到时先重组信息层级，再调整排版。
8. 确认没有禁用栏目、表情符号、科技海报式装饰图、非 ImageGen 自定义图和未确认事实。
9. 从大厂架构师视角静默评审一次：主板/设备形态、职责边界、跨端数据流、协议一致性、资源约束、恢复路径、批量交付口径和面试追问能否互相对上。默认直接修正文，不附独立评审表。
10. 存在真实面经时，确认主稿中的正式问题和嵌套追问都有灰色来源行与完整紫色回答；同答案逻辑已经合并，原题档案入口可访问，主稿没有复制原题列表，也没有把面经问题当成项目事实。
11. 复读独立原题档案，确认记录数、原题数和来源数可复算，全部在范围内的原题保持原顺序，重复、粗糙表述和错误前提没有被擅自删除或改写。

## 版本更新

- 飞书可用时，写入前后记录 revision；离线回退时，记录 DOCX 的起止版本。交付时说明版本变化和实际改动范围。
- 文档结构、技术事实或口述回答变化时提升 DOCX 版本号；仅修正错字或版式时提升补丁版本。
- 飞书可用时，生成 DOCX 前再次读取最新内容，并确认快照与该 revision 一致；离线回退时，渲染最新 DOCX 并抽查目录、图片、颜色、分页和链接。
- 正常模式下，DOCX 发布后继续迭代时以飞书的新 revision 为起点重新导出。离线回退结束后，只做一次“最新 DOCX → 飞书活稿”迁移并核对内容，此后不再从旧 DOCX 反向合并修改。

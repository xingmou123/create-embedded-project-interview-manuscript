# 项目面试手稿的样式与交付规则

## 文档骨架

所有项目沿用同一套外层骨架，技术章节按项目实际内容增减：

1. 项目定位：解决什么现场问题，设备边界是什么，候选人负责什么。
2. 30 秒与 90 秒讲法：先给短版本，再给完整主线。
3. 架构、执行域、数据流和职责边界：先分清谁采、谁算、谁传、谁恢复。
4. 技术问答：按真实数据流组织，不按简历条目机械拆章。
5. 实战调试案例：案例边界来自项目资料，缺失的机制、参数和故障注入证据方案按项目底稿补全，并形成完整因果链。
6. 设计取舍与反向追问：解释为什么这样做，以及哪些边界不能夸大。

技术章节通常按“设备与执行域 → 采集与驱动 → RTOS 与数据流 → 算法与状态判断 → 对外通信 → 维护升级 → 故障诊断”展开。项目边界必须有资料支撑；边界内缺失的机制和数值按项目底稿完成工程补全。例如项目没有 BLE 或 Bootloader，就不要为了结构完整硬加章节。

如果处理器、执行域或主辅链路仅靠文字不容易分清，在 30/90 秒口述之后、技术问答之前放一张整机图；简单项目不强制配图。

## 问答单元

每个问题保持以下结构：

```xml
<h2>Q1：问题</h2>
<p><b><span text-color="rgb(100,37,208)">可直接对面试官口述的回答。</span></b></p>
<p><b>学员备注：</b>只解释真正容易混淆、且口述回答不适合展开的内容。</p>
```

- 紫色加粗段只放口述回答。写成自然说话，不加栏目提示，不在里面穿插教学说明。
- 回答先说结论，再讲关键机制、边界或证据。能一段说清就不要拆成三段。
- 黑色学员备注按需添加，不是每题必配。通常一到两小段即可。
- 术语第一次出现且可能卡住学员时，用一句白话解释；不要重复教材定义。
- 数据手册、协议规范和工具链接放在学员备注，不塞进口述回答。
- 调试案例按“现象 → 排查范围 → 关键证据 → 根因 → 修改 → 验证”自然讲述，但不要把这六项写成机械标签。

## 语言要求

- 写具体对象、动作、时序和结果，例如“HT 和 TC 连续到来，二值信号量只保留一次唤醒”，不要写空泛的“提升系统能力”。
- 句子短，主语明确。读出声时应像工程师在复盘项目，不像背报告。
- 芯片、接口、参数和指标先服从项目口径底稿。资料缺失时，允许按官方约束和成熟量产实践完成明确的工程选择；先写入底稿并核对全篇，不能在某道题里临时另造一套参数。命名客户、认证、出货量、历史故障和个人职责等事实仍须有来源或用户确认。
- 技术回答要能承受追问。通信校验、业务正确性、缓冲所有权、掉电恢复等概念不要混为一谈。
- 不堆“首先、其次、最后”，不使用“核心能力、关键抓手、形成闭环、进一步赋能”等抽象套话。
- 禁用“面试官意图”“核心亮点”“标准答案”“自测题”等栏目。
- 不使用表情符号、装饰性提示框、花哨分栏和无信息价值的强调。

## 图像规则

图片按以下顺序选择：

1. 用户提供的真实照片、示波器波形、日志、界面截图或板卡材料。
2. 已有软硬件设计稿中的架构图、时序图、状态图和接口图。复用前必须核对是否与当前手稿一致。
3. 为确定关系制作的简化 SVG 图。只画已经确认的模块、数据方向、状态和时序，不补想象中的结构。
4. 生成式图片只用于不要求技术精度的氛围图。项目面试手稿通常不需要这类图片。

优先为以下内容配图：多处理器边界、跨任务数据所有权、DMA 时序、协议分帧、升级状态机和故障证据路径。单一结论能用两句话说清时，不额外画图。

每张图都要有编号和标题。图中的芯片名、接口名、方向、单位、状态名必须和正文一致。出现“待定”“待固化”或与正文冲突的旧图时，先修图再交付，不能靠正文解释过去。

## 飞书与 DOCX 的交付关系

- 迭代阶段以飞书文档为唯一活稿。问答、图片、备注和顺序都在飞书中更新。
- 内容确认后默认同时交付飞书活稿和从同一 revision 生成的 DOCX 快照。DOCX 不与飞书同时维护。
- 飞书可用时，DOCX 交付后如需修改，先回到飞书更新，再重新生成一个新版本的 DOCX。不要分别修改两份文件。
- 飞书可用时，旧 DOCX 只保留归档，不再继续编辑。离线回退时只把最新版本 DOCX 作为临时活稿，每次改版归档上一个版本。文件名使用 `项目名_项目面试手稿_vX.Y_YYYYMMDD.docx`。
- 飞书暂时不可用时，以 DOCX 作为临时主稿并记录版本；飞书恢复后将最新 DOCX 建成飞书活稿，再按上述单活稿规则迭代。用户明确只要一种格式时，按用户要求交付并说明没有生成另一份。

## 写入前检查

1. 飞书可用时，读取最新 revision、目录和待改章节，任何写操作都以最新 block id 为准；离线回退时，记录最新 DOCX 的文件名、版本和修改时间，并只在该版本上继续。
2. 对照简历、设计文档、代码或用户补充材料，确认项目边界、处理器数量、总线用途和实现成熟度。
3. 检查已有图片中的接口、参数、方向和状态是否已经过时。
4. 标出缺少证据的内容。技术机制和参数按项目底稿完成工程补全；涉及命名客户、认证、出货、个人职责或真实历史的内容先向用户确认，不能自行写成事实。
5. 确定本次只新增、替换或移动哪些块，避免顺手改动无关章节。

## 写入后检查

1. 重新读取目录和所有改动章节，检查顺序、重复块、标题层级和段落位置。
2. 确认口述回答仍为紫色加粗，学员备注为普通黑字，没有把备注混进口述段。
3. 顺着整机数据流通读一遍，检查执行域、缓冲所有权、协议边界、升级状态和调试证据能否互相对上。
4. 逐张核对图文，特别检查接口名、单位、箭头、协议时序、DMA 事件和存储分区等容易误导的内容。
5. 把每段紫色回答实际读一遍；不顺口、句子过长或听起来像报告时，继续压缩。
6. 确认没有禁用栏目、表情符号、装饰图和未确认事实。

## 版本更新

- 飞书可用时，写入前后记录 revision；离线回退时，记录 DOCX 的起止版本。交付时说明版本变化和实际改动范围。
- 文档结构、技术事实或口述回答变化时提升 DOCX 版本号；仅修正错字或版式时提升补丁版本。
- 飞书可用时，生成 DOCX 前再次读取最新内容，并确认快照与该 revision 一致；离线回退时，渲染最新 DOCX 并抽查目录、图片、颜色、分页和链接。
- 正常模式下，DOCX 发布后继续迭代时以飞书的新 revision 为起点重新导出。离线回退结束后，只做一次“最新 DOCX → 飞书活稿”迁移并核对内容，此后不再从旧 DOCX 反向合并修改。

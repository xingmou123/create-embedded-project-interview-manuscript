## 1. 完成门槛

每份完整项目面试手稿必须交付**正好三个**深度实战调试案例。少于三个不完整，多于三个会稀释重点；其余候选只留在内部底稿，不进入学员版。

三个案例逐个按证据分类为“已证实真实”“故障注入”或“设计推演”。只有代码、日志、测试记录、可复现测量或用户对具体项目的明确确认支持时，才能标为已证实真实并使用过去时复盘；其余两类必须使用与事实相符的语态，不能暗示实际发生。案例仍要建立完整证据链，也不能借此声称读取过不存在的日志、波形或测试报告，更不能添加命名客户、现场日期、仪器序列号、测试编号或未经资料确认的个人职责。

案例必须从项目实际架构和数据流中生长出来；有简历或职责材料时，还必须落在候选人的职责范围内。不能先套“驱动、RTOS、通信”三类模板，再替换芯片名。穿戴设备可以落在低功耗、BLE 重连或传感器漂移；机器人可以落在控制抖动、编码器异常或 CAN 丢帧；AI 终端可以落在内存峰值、任务阻塞、缓存一致性或网络恢复。具体选择始终以本项目主线为准。

三个案例共同回答一件事：候选人面对复杂故障时，能否根据现象收集证据、缩小范围、找到根因，并用可验证的修改解决问题。

## 2. 先建候选池，再选三个

### 2.1 建立 6～10 个候选故障

根据项目口径底稿逐段扫描以下位置，列出 6～10 个候选：

- 传感器或执行器与 MCU 之间的数据边界；
- 中断、DMA、缓存、任务和协议之间的交接位置；
- 时序、并发、资源耗尽、掉电、异常恢复等边界条件；
- 算法输入质量、状态机切换、阈值与业务告警之间的语义边界；
- 现场通信、升级、存储、低功耗、热管理或网络恢复等项目特有路径；
- 简历中由候选人负责、且面试官容易连续追问的部分。

每个候选先写一行因果骨架：

`可观察现象 → 受影响的数据路径 → 可采集证据 → 候选根因 → 对应修改 → 验证方法`

如果其中任意一段只能靠一句“可能是”连接，先降级为备选，不要直接扩写。

### 2.2 动态评分

按下表直接打分，单个候选总分为 0～90 分：

| 单候选评分项 | 满分 | 判断标准 |
|---|---:|---|
| 与项目主线的相关度 | 25 | 是否直接经过核心数据流、控制流或交付功能 |
| 技术深度 | 20 | 是否涉及可解释的底层机制，而不是普通配置错误 |
| 证据与因果链完整度 | 20 | 能否用日志、计数器、波形、寄存器或故障注入把范围收窄 |
| 面试追问价值 | 15 | 是否能自然展开原理、选型、边界与取舍 |
| 与候选人职责的匹配度 | 10 | 是否属于简历中可解释、可承担的工作范围 |

单个候选满分 90 分。先选出 4～6 个高分候选，再枚举合理的三案例组合。每个组合使用三个候选的平均分，加 0～10 分的组合差异度，得到 100 分制结果；差异度只在这里计算一次。不要机械取单项分数最高的三个，组合还必须通过下面的去重门槛。

缺少简历或职责原文时，“职责匹配度”记为 N/A，不得擅自填中间分。此时单候选按其余四项的 80 分排序，组合再加 0～10 分差异度，架构覆盖模式总分最高 90 分；该分数不与正常模式的 100 分制横向比较。拿到职责材料后再补职责分并复排。案例性质和候选人具体负责哪一步分别取证，不能由技术合理性相互推出。

### 2.3 去重门槛

满足下列任一情况，应视为重复并替换其中一个：

- 只是把同一种超时、丢包、越界或配置错误换了芯片名；
- 三个案例都靠加日志发现，没有波形、计数器、寄存器、内存或时序方面的不同证据；
- 根因都可以概括成“任务来不及处理”；
- 修改都只是“增加延时、扩大缓冲区或提高优先级”；
- 三个案例集中在同一条局部路径，无法体现对整机的理解。

差异不是按模块名称判断，而是按**失效机理、证据类型、修复方法和验证方式**判断。

## 3. 单个案例的七项硬结构

每个案例必须包含以下七项，顺序可以为口述自然性做小幅调整，但不能缺项。

### 3.1 具体现象与受影响的数据路径

说清楚在什么运行条件下，哪个可观察量出现了什么异常，以及异常沿哪条路径影响了最终功能。避免“系统偶发异常”“数据不稳定”这类空话。

至少交代：触发条件、正常表现、异常表现、出现频率或边界、受影响和未受影响的模块。

### 3.2 缩小范围与排除过程

先说明最初有哪些合理怀疑，再说明如何逐个排除。排查顺序要符合成本和证据强度：先确认现象和输入，再切分软硬件边界，最后深入并发、时序或器件内部行为。

不能直接从现象跳到根因，也不能用“凭经验判断”代替排查。

### 3.3 关键证据及工具的实际作用

仪器、日志和调试工具必须真正参与推理。写清楚：

- 测了什么信号、寄存器、计数器或内存；
- 在什么时刻采集；
- 观察到什么差异；
- 这条证据排除了什么、支持了什么。

示波器、逻辑分析仪、J-Link/RTT、Ozone、Trace、异常快照、协议抓包、GPIO 打点、性能计数器等只在有实际作用时出现。禁止为了显得真实而罗列仪器，也不要虚构具体仪器型号、客户现场、批次、日期或测试编号。

### 3.4 完整根因链

根因要从底层行为一直解释到业务现象，中间不能断层。推荐写成：

`触发条件 → 底层机制 → 中间状态/数据被破坏 → 上层误判或失效`

例如，“CRC 正确但业务值错误”必须说明错误发生在 CRC 计算之前；“偶发 HardFault”必须落到具体非法访问、栈破坏、竞态或异常上下文，不能停留在“通信超时导致崩溃”。

### 3.5 与根因一一对应的修改

修改必须直接切断根因链，而不是掩盖现象。需要写清：

- 改了哪个状态机、所有权、时序、缓冲策略、校验或恢复流程；
- 中断与任务各自保留什么职责；
- 异常数据如何标记、丢弃或恢复；
- 为什么没有选择看似简单的临时办法。

如果修改只剩“加延时、加锁、扩大数组、提高任务优先级”，通常说明根因还没有解释清楚。

### 3.6 针对原故障的验证

验证必须攻击原来的失效条件，至少包含两类：

- 原场景回归：证明原故障不再出现；
- 边界或故障注入：主动制造超时、积压、断连、丢帧、掉电、内存压力等条件；
- 长稳运行：观察错误计数、资源占用、恢复次数和业务输出；
- 兼容性或交叉验证：不同设备、波特率、负载、环境或版本下复测。

不要只写“长时间测试正常”。需要说明观察指标，以及系统在故障仍会发生时如何安全降级、记录和恢复。

### 3.7 必要的补充理解

只有以下情况才加补充理解：器件行为容易记错、概念容易混淆、面试官可能追问边界、完整回答为简洁而省略了关键前提。

补充理解用普通话解释原理、数字来源和不能说错的边界。它不是第二份答案，不重复完整回答，不增加自测题。简单案例可以没有补充理解。

## 4. 证据—根因—修复—验证必须闭合

扩写前先做一次四列表，任何空格都代表案例还不能交付：

| 证据 | 它证明或排除了什么 | 对应根因 | 修改与验证如何命中根因 |
|---|---|---|---|
| 具体波形、日志、计数器、寄存器或快照 | 写出推理作用 | 写到底层机制和上层后果 | 写出切断点及复现条件下的结果 |

闭合关系必须满足：

1. 每个根因至少有一条正向证据，并有必要的排除证据；
2. 每项修改都能指出它切断了根因链的哪一环；
3. 每项验证都覆盖原触发条件，而不是只验证正常路径；
4. 如果故障不能完全消除，必须验证检测、隔离、降级和恢复是否生效；
5. 业务结果与底层指标一致，不能只看“没有报警”或“没有死机”。

## 5. 事实边界

案例服从项目口径底稿的事实优先级，不得反向修改已经锁定的主 MCU、接口、器件、任务关系、协议或职责范围。

- 有代码、原理图、日志、测试记录或用户确认时，严格按证据写；
- 三个案例逐个记录证据和性质；资料缺失的机制、参数和验证方法可以依据官方手册、协议规范和成熟工程实践补全，但只能作为故障注入或设计推演，且不得捏造具体日志原文、波形读数或测试报告编号；内部底稿必须标明依据并保持技术可复核；
- 工程补全不能添加命名客户、认证结果、现场日期、仪器序列号、测试编号、团队规模或个人职责；
- 不为凑案例新增芯片、无线方式、外设、协议或业务功能；
- 器件寄存器行为、协议时序和 RTOS/API 语义应由官方资料核对；
- 用户限制来源范围或网络不可用时，未完成官方核对的器件行为留在内部待核对表；依赖该行为的候选不能进入最终三个；
- 数值必须与整份手稿一致，并能解释计算方式、余量和适用边界。

如果案例与项目其他章节冲突，先修订内部口径，再统一修改全部引用位置。禁止只改案例使其局部自洽。

## 6. 交付写法

### 6.1 口述版

口述版应像工程师复盘问题，能直接说给面试官听。推荐节奏：

1. 一句话说现象和影响；
2. 两三句说明如何复现、切分范围；
3. 用关键证据引出根因；
4. 解释完整因果链；
5. 说清修改和为什么这样改；
6. 用回归、故障注入或长稳结果收尾；
7. 最后只留一句真正有价值的工程认识。

口述语态服从逐案证据。已证实真实的案例可以写“我们当时遇到……我先看了……后来把……改成……”；故障注入写清“我们在测试中主动注入……”；设计推演写清“如果出现这种现象，我会先……”。不能为了听起来像实战而把后两类改成真实历史。

三种语态都要使用具体动作。句子宜短，不堆术语，不用“首先、其次、最后”机械串联，也不用“体现了系统性思维”等评价自己。关键技术词必须能被后续追问，不能只作为装饰。

### 6.2 单页阅读层级

学员第一次看到案例时，应在十秒内找到四个答案：出了什么问题、根因是什么、怎样修改、是否验证通过。按以下规则排版：

- 标题直接写故障矛盾，不使用“案例一”“高频题”等空标题；
- 紫色主回答先给一句结论，再用不超过三段完成“现象与范围、证据与根因、修改与验证”，每段两三句话；
- 七项硬结构用于作者审查，不得把“现象、拟执行、判断条件、证据成立时、确认后、验收计划”等内部标签机械展示给学员；
- 主回答通常只保留二至四个决定性数字，其余波特率、任务名、字段名和压力测试明细下沉到黑色补充理解；
- 需要解释争用、状态或时序时，增加一张纯 ImageGen 的修复前后对照图；能用两句话讲清时不凑图；
- 不使用表情符号、黄色提示框或培训课件式装饰。

### 6.3 补充理解

补充理解放在完整回答之后，与正文视觉上区分。内容优先包括：

- 一个容易说错的器件或协议事实；
- 一个“为什么不是另一种方案”的简短解释；
- 一个面试官继续追问时必须守住的边界；
- 一个数值的计算依据或适用条件。

补充理解仍使用简洁工程语言，不写“面试官想考察”“标准答案”“核心亮点”，不做题库式扩展。

## 7. 跨项目选题提示

以下仅用于寻找候选，不代表固定分类或固定配额：

- 穿戴设备：睡眠电流异常、BLE 重连风暴、传感器运动伪影、FIFO 溢出、功耗状态切换竞态；
- 机器人：控制周期抖动、编码器计数异常、CAN bus-off 恢复、PWM/ADC 同步、电机堵转误判；
- AI/视觉终端：摄像头 DMA 与 Cache 一致性、内存碎片或峰值、推理任务饿死、热降频、网络断线恢复；
- 工业终端：采集窗口覆盖、Modbus 帧边界、现场干扰、升级掉电恢复、告警语义错误；
- 医疗或仪器设备：采样同步、前端饱和恢复、数据完整性、校准漂移、故障安全状态。

项目没有某条路径时，不得为了套用这些例子把它补进去。

## 8. 独立技术审查清单

三个案例写完后，必须由独立审查者在不参考作者推理过程的情况下逐项检查。优先交给一个新子代理，只提供原始项目资料、项目口径底稿和三个成稿案例，不提供作者的评分结论；子代理不可用时，隔离作者草稿，按本清单重新核对官方资料和因果链。任意一项不通过，退回修改：

- [ ] 交付中正好三个案例，另有 6～10 个内部候选及评分记录；
- [ ] 三个案例均已逐个记录证据和性质，口述语态与“已证实真实/故障注入/设计推演”一致；
- [ ] 三个案例来自项目主线；有职责材料时同时检查职责范围，架构覆盖模式下职责项记 N/A 且不阻塞；
- [ ] 三个案例的失效机理、证据类型、修复方式不重复；
- [ ] 每个案例都具备七项硬结构；
- [ ] 现象能够由根因完整推导，中间没有机制断层；
- [ ] 排查过程有合理的候选假设和排除证据，没有直接猜中答案；
- [ ] 日志、计数器和仪器分别说明了测量对象、时刻、结果及推理作用；
- [ ] 没有装饰性仪器、虚构客户历史或无来源的具体测试记录；
- [ ] 器件、协议、RTOS、编译器、Cache、DMA 等关键行为已经按官方资料核对；
- [ ] 修改直接命中根因，并处理了异常数据、恢复路径和资源所有权；
- [ ] 验证重现了原触发条件，包含边界或故障注入，并观察了明确指标；
- [ ] 案例中的接口、数值、任务、芯片和整份手稿口径一致；
- [ ] 口述版可以自然说出，不像论文、事故报告或 AI 模板；
- [ ] 学员能在十秒内找到现象、根因、修改和验证，主回答没有暴露内部推演标签；
- [ ] 补充理解只解释难点和边界，没有重复正文或增加自测题；
- [ ] 每个案例至少能自然连接到一处前文技术问答，形成“原理回答—实战证据”的呼应。

以下任一情况直接判定不合格：直接跳到根因；修复与根因不匹配；三个案例只是同一问题换名；工具没有参与推理；器件或协议事实错误；验证没有覆盖原故障；为了故事完整而改写项目既有事实。

# 手稿封版与简历项目库归档

## 触发条件

仅在用户明确确认当前手稿已经完成，或明确要求存入简历项目库时执行。用户只是要求审计、修改、补图或继续完善时，不提前归档。

## 归档目标

简历项目库只保存项目描述和手稿入口，不复制整份手稿正文。每个项目只保留一个可用的“项目手稿”入口，并指向该项目当前唯一飞书活稿。

## 执行顺序

1. **锁定活稿**：按项目名、文档 token、完整目录、封版记录和最新 revision 确认唯一活稿。编辑时间只能辅助判断，不能单独决定哪个同名文档是最终稿。
2. **锁定项目库**：优先使用用户给出的项目库链接、当前主线程已经冻结的项目库 token，或包含目标项目且“项目手稿”正指向该项目已知旧活稿的库页。只有“采用统一字段”“包含同名项目”或“编辑时间较新”不能证明它是活库；多个候选仍无法由明确 token、主线程记录或旧链接对应关系唯一裁决时，向用户只问一次写入目标。
3. **定位项目条目**：按完整项目名搜索，读取项目标题、任职时间、“项目手稿”段落和紧邻的项目简介/描述。不得仅凭相似关键词修改其他项目。
4. **最小写入**：
   - 已有“项目手稿”且指向当前活稿：不重复写入；
   - 已有“项目手稿”但指向旧稿：只替换该段落的链接；
   - 没有“项目手稿”：沿用项目库现有样式，在项目名称或任职时间之后、项目简介/描述之前插入 `<p><b>项目手稿：</b><a href="活稿链接">打开手稿</a></p>`；
   - 不修改项目名称、任职时间、项目简介、工作内容或相邻项目。
5. **写入保护**：写前读取最新 revision 和目标 block ID。无论入口是 `/docx/` 还是 `/wiki/`，正文都使用飞书文档的局部 block 操作；`lark-wiki` 只用于解析节点身份和层级，不得为了登记手稿链接而移动、复制或创建 Wiki 节点。不得创建第二份项目库、第二个项目条目或手稿副本。
6. **复读验证**：写后重新读取受影响区域，确认项目名、任职时间、手稿链接和项目简介/描述顺序正确；再读取链接目标，确认 token、标题和一级目录属于刚完成的活稿。检查同一项目附近没有第二个“项目手稿”入口，也没有旧链接残留。

## 交付口径

归档成功后只说明三件事：更新了哪一份项目库、归档的是哪一份活稿、旧链接是否已替换。分别给出项目库和手稿链接。不要把搜索过程、候选文档、内部 token 比对或制作痕迹写进学员可见文档。

## 禁止事项

- 不因存在多个同名手稿就全部登记；
- 不因存在多个同名项目库就全部修改；
- 不把旧稿和新稿并列为两个“最终版”；
- 不把手稿全文粘贴进项目库；
- 不覆盖或移动项目描述；
- 不在没有用户封版确认时自动归档尚在修改的活稿。

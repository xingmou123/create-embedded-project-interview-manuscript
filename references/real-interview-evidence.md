# 真实面经：证据、正式回答与原题档案

当任务涉及检索、整理或使用真实面经时，按本文件执行。真实面经只能证明“面试官问过什么”，不能证明当前项目采用、实现或验证过题目里的方案、参数和经历。

可靠来源必须能看出真实面试上下文，例如公司或岗位、轮次或时间、完整复盘、录音或逐字整理，并且原文可访问复读。课程题库、教程、模拟面试、教师补答、参考话术、缺少上下文的匿名拼接和无法复读原文的截图，不作为真实面经来源。

## 三种材料不要混在一起

- **内部证据表**：记录每份面经、每道原题、来源链接和正式问题映射，只用于查漏和核验。
- **主稿的真实面试问答**：把反复出现、真正涉及当前项目的问题整理成学员能直接练习的正式问题，每题配一份完整口述回答。
- **独立原题档案**：按原面试记录保存全部在范围内的原题和来源，不补答案，不做教学改写。它是溯源资料，不是第二份主手稿。

不要把“精选题”“训练题”和“原题档案”混成一份文档。主稿负责学，原题档案负责保真和压力测试，内部证据表负责证明两边没有漏。

## 确定收录范围

面经记录明确讨论过当前项目、同一份项目经历或该项目的职责时，整份记录纳入原题档案。即使其中夹有 C 语言、操作系统、网络或算法等通用题，也按原顺序保留，因为它们属于这场真实面试的完整追问环境。

只与相似技术栈有关、但没有讨论当前项目的面经，可以用于发现主稿缺口；默认不混入“涉及本项目的全部原题”统计。用户明确要求扩大范围时，再另行标记来源范围，不能把两类数量合并后声称都直接命中本项目。

## 建立内部证据表

为每份面经分配记录 ID，为记录中的每道题分配原题 ID，至少保存：

| 字段 | 内容 |
|---|---|
| 记录 ID | 一场面试或一份完整复盘的唯一编号 |
| 公司、岗位、轮次 | 原文能确认多少就写多少，不猜测 |
| 来源链接 | 可访问的原始飞书、妙记或其他来源 |
| 原题 ID 与顺序 | 保持原记录中的出现顺序 |
| 原题原文 | 保留口语、重复和不完整表达，不润色 |
| 与项目的关系 | 直接项目追问 / 同场通用题 / 相似技术栈补充 |
| 正式问题 ID | 映射到主稿中的哪一道规范问题 |
| 处理结果 | 新增回答 / 补强回答 / 合并表述 / 仅存档 |

统计“出现次数”时，以不同面经记录的数量为准，不把同一场复盘里的重复问法重复计数。来源无法访问、面试上下文无法确认或记录重复转载时，先继续查证；最终仍无法确认的材料不计入真实面经数量。

## 从原题归并为正式问题

归并依据是“完整回答是否相同”，不是文字看起来是否相似：

1. 多种问法可以用同一套项目事实、实现路径、取舍和边界回答时，合并成一道自然的面试问题。
2. 追问引入了新的故障条件、设计选择或职责边界，必须给出不同回答时，保留为该题下的嵌套追问。
3. 只是换公司、换说法、追问同一个结论，或完整回答已经自然覆盖的句子，不另起一题。
4. 不同行业或公司只有在约束变化会实质改变答案时才保留差异追问；否则只体现在来源行里，不单独制造行业版本。
5. 前提错误、版本过时或与项目口径不一致的原题仍保留在原题档案，但不得把错误前提带进正式回答；内部证据表把它标为“仅存档”。

正式问题的数量不预设。以核心项目问题都能独立回答、相同答案没有被拆成多题、不同答案没有被硬合并为停止条件。

## 主稿必须给一份完整回答

主稿在技术学习章节之后增加“真实面试问题与完整回答”一章，作为学员已经理解项目后的考前突击。问题按当前项目最容易复习的技术主线组织，不按公司逐家堆题。

每个可见问题只对应一份完整回答：

- 不生成“30 秒版、90 秒版、简版、进阶版、参考版”等多个答案版本；项目介绍章节原有的 30/90 秒介绍不受此规则影响。
- 回答写成一段连贯、口语化、可以直接讲给面试官听的紫色正文；确因信息密度需要分段时，仍是一份连续回答，不拆成多个方案。
- 先自然说结论，再结合本项目讲清实现过程、选择原因、异常处理、验证证据和能力边界；不是每题都机械凑齐这些项，也不把它们写成小标题。
- 不显示“思路、实现、原因、异常、标准答案、面试官意图、评分、通过标准、必须讲透、压力追问、连续追问”等教学标签。
- 只有概念确实容易混淆、且塞进主回答会破坏口述流畅性时，才增加一小段普通黑色“补充理解”。

真实追问采用标题层级自然嵌套：主问题使用当前章节的主问题层级，追问降一级。不要另写“下面是连续追问”或用列表堆一串没有答案的问题。每个保留下来的追问都必须紧跟自己的完整紫色回答。

## 来源行

每道正式问题在标题下、回答前放一行简短灰色来源，格式优先为：

`真实面经：经纬恒润、卓驭科技等 3 份复盘出现。`

只出现一次时可以写：

`真实面经：某公司一面，1 份复盘出现。`

来源行列出有代表性的公司或岗位，并给出不同面经记录的出现次数。公司过多时使用“等”，不要铺满一行；不要把公司差异解释成答案差异。能稳定链接到原题档案中的对应记录时，可把公司名或“查看原题”做成链接；不能稳定定位时，使用本章统一的原题档案入口，不堆裸链接。

## 独立原题档案

飞书可用时，原题档案必须是独立飞书参考文档，尽量与主稿放在同一项目目录。它不是第二份主手稿，不参与主稿 revision 竞争，也不默认导出 DOCX；飞书不可用时才使用独立的本地参考文件。

原题档案按“面经记录”组织，优先使用公司、岗位、轮次或记录 ID 作为标题。每份记录只做三件事：

1. 写清原文能确认的公司、岗位、轮次或时间；
2. 放置可访问的原始来源链接；
3. 按原顺序逐题保存原文。

一旦整份面经记录进入范围，就不得因为重复、口语粗糙、价值低、前提错误或暂时没有答案而删题、改题或重新排序。原题档案不补生成答案，也不复制主稿答案。需要教学归并、纠错或取舍的内容只在内部证据表和主稿中处理。

档案开头写明本次核验后的面经记录数、原题数和可访问来源数。计数口径固定后再写入，不能用大约、近百道等模糊表述替代可核验数量。

## 写入顺序

存在大量原题，或现有主稿已经被原题附录拖得过长时，严格按以下顺序处理：

1. 读取主稿和全部候选来源的最新 revision，建立内部证据表并冻结收录范围。
2. 先创建或更新独立原题档案，写入全部在范围内的记录、原题和来源。
3. 复读原题档案，核对记录数、原题数、来源数、题目顺序和链接可访问性。
4. 再根据答案逻辑归并正式问题，补写或修订主稿中的完整紫色回答和灰色来源行。
5. 最后把主稿里的大段原题附录替换成一个紧凑的原题档案入口；在档案未通过核验前，不得删除主稿中的旧原题。
6. 分别复读两个文档的受影响章节和相邻段落，确认主稿仍是唯一活稿，原题档案明确是参考文档。

主稿中的入口只需要标题、一小段用途说明和一个可访问的文档卡片或书签。不要再次复制原题列表。

完整交付时，在项目 `work` 目录保存 `real_interview_package.json`，记录主稿检查结果、档案计数、每份记录、每道原题和每个正式问题。核心字段如下：

## 交付检查

- 每份在范围内的真实面经都有记录 ID、可访问来源和完整原题顺序。
- 原题档案的记录数、原题数和来源数能由内部证据表复算。
- 每道进入主稿的正式问题都有且只有一份完整紫色回答和一行灰色来源。
- 相同答案逻辑已经合并；保留的嵌套追问确实需要不同回答，并且每题都有答案。
- 行业或公司差异只在会改变答案时成为独立追问。
- 部分覆盖和未覆盖的核心问题已经反向修订正式问答。
- 主稿没有教学优先级、评分、筛选过程、多版本答案或成串裸题。
- 原题档案没有擅自删题、润色、去重、补答案或把面经内容冒充项目事实。
- 主稿只保留紧凑的原题档案入口，没有复制完整原题库。
- 工程经验补出的值统一写作“设计参数”，没有冒充实测结果。
